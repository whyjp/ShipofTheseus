### 알고리즘 검증

ⓔ **인터럽트 0 검증** (`self_lint C23`) — phase 05~13 본문에 ack 호출 없음 lint.

### 임계 도달 보장 분석

**임계 0.999 / 0.99999 가 *반드시* 도달한다는 보장은 없다** — 다음 케이스에 fallback:

ⓐ 천정 도달 + Q-D3 답 = `4 (정체 수용)` → 게이트 영구 fail 로 표시, 다음 페이즈 진행 (임계 미달 종료 가능).
ⓑ 정체 누적 3 회 + Q-D4 답 = `3 (정체 수용)` → 차원 게이트 비활성, 다음 페이즈 진행.
ⓒ DIP 위반 모든 멀티버스 우주 → `halt_for_intent_mismatch` (의도 모순으로 정지).

이게 *합리적 자율의 한계* — 본 스킬은 무한 시도가 아닌 *합리적 자율*. 사용자가 페이즈 04 에서 위임 수준을 *최대 자율* (옵션 1) 로 답하면 임계 미달 상태로도 핸드오프까지 가능하지만, 그 사실이 산출물에 명시 기록.

### tree 기반 시도 (AIDE)

ⓐ checkpoints.md + checkpoint.py 의 회귀 매핑 = AIDE 백트랙.
ⓑ lesson_pack 의 누적 = AIDE Memory.
ⓒ 멀티버스 N 우주 = tree 의 분기.

**완전 구현됨**. 단, 실제 외부 프로젝트에서 tree 가 얼마나 깊어질지는 미측정.

## 4. 파편화 필요와 그레이드 시스템 — 가장 큰 갭

### 현재 상태의 문제

본 하네스는 *모든 호출* 에 14 페이즈 풀 진행 + 26 컨벤션 적용을 가정. 이는:

ⓐ **단순 작업에 over-engineering** — 단일 함수 리네임에 명명/의도/리뷰/콜드 재이해/질의/비평/계획/재계획/구현/게이트/스프린트/회귀/웹뷰/핸드오프 14 페이즈 진행은 비용 폭발.
ⓑ **사용자 호출 부담** — 사용자가 호출 *전* 에 "내 작업이 본 하네스 가치 있는가" 판단 불가.
ⓒ **호출 거부 룰 부재** — SKILL.md 의 "호출 금지" 섹션이 한 줄 수정/리네임/throwaway 만 나열, *경계가 모호한* 작업의 그레이드 매핑 없음.

### 필요한 신규 — `conventions/grades.md`

5 그레이드 + 적용 컨벤션 매트릭스 + 페이즈 04 첫 질의로 그레이드 확정.

| Grade | 작업 예 | 페이즈 | 컨벤션 수 | 멀티버스 |
| ----- | ------ | ------ | --------: | -------- |
| 1. Trivial | 한 줄 수정 | 호출 거부 | 0 | X |

### 그레이드 판정 알고리즘

ⓐ **자동 추정** (`scoring/grade_assess.py`) — 원문에서 도메인 키워드, 모듈 수 추정, FE/BE 여부, "결제/금융" 등 키워드 → 후보 그레이드 1~3 개 산출.

ⓒ **그레이드 다운그레이드 권고** — 사용자가 잘못된 그레이드 (예: 단순 작업에 Grade 4) 호출 시 본 하네스가 자율 다운그레이드 권고 + 사용자 객관식 ack.

### 허들 구조

본 하네스 호출 *직후* 첫 동작 = 그레이드 판정. Trivial 이면 "본 하네스 호출 거부, 단순 응답 권고" 메시지로 마무리. 그 외는 그레이드별 페이즈/컨벤션 활성화.
