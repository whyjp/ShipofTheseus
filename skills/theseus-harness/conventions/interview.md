# 인터뷰 컨벤션

## 한 줄 요약
**모든 사용자 질의는 두괄식 + 한 번에 하나 + 객관식 5개 이하 + 보기는 무조건 숫자 + 제안과 선택을 같은 블록에 섞지 않는다.** 이 규칙은 본 하네스가 사용자에게 던지는 모든 질문에 예외 없이 적용된다.

## 상세

### 1. 두괄식
질의의 첫 문단은 **요약 한 문장**. 무엇을 묻고, 왜 묻는지 한 줄로 결론. 배경·트레이드오프·고려사항은 그 다음 문단으로 분리한다. 사용자가 첫 줄만 읽고도 답할 수 있어야 한다.

### 2. 1회 1질의
한 호출에 한 질문만. 다음 질문은 답을 받은 뒤. 묶음 질문은 사용자가 답하기 전에 의식 부담을 키우고, 첫 답이 두 번째 답을 흐리게 만든다.

### 3. 객관식 5개 이하
선택지가 5개를 넘으면 잘못 모델링한 것이다. 비슷한 옵션을 합치거나, 직교 차원으로 쪼개 두 단계로 나눠 묻는다. "기타" 는 마지막 옵션으로 둘 수 있고 자유 응답을 받는다.

### 4. 보기는 무조건 숫자
객관식 선택지는 `1`, `2`, `3`, `4`, `5`. 알파벳·기호 사용 금지. 사용자가 "3" 이라고만 답해도 명확해야 한다.

### 5. 서브 옵션은 `숫자-알파벳`
숫자 보기 안에 더 세분이 필요하면 `1-a`, `1-b`, `1-c`. 깊이는 한 단계까지만 — 더 깊어지면 별개 질의로 분리한다.

### 6. 순차 리스트(선택지 아님)는 원형 유니코드
질문이 아닌 **설명 안의 순차 항목** — 단계, 근거 나열, 예시 — 은 원형 알파벳 우선:

ⓐ 첫 항목
ⓑ 두 번째
ⓒ 세 번째

숫자가 꼭 필요하면 원형 숫자로:

① 첫 단계
② 두 번째 단계
③ 세 번째 단계

순수 텍스트 숫자(`1.`, `2.`)는 객관식 보기에만 쓴다 — 두 용도가 시각적으로 섞이면 안 된다.

### 7. 제안과 선택의 분리
"이런 안이 가능하다(제안)" 와 "둘 중 무엇으로 갈까요?(선택)" 를 같은 블록에 섞지 않는다. 문단·헤더로 분리하고, 선택 블록만 객관식 형식을 쓴다. 제안 단락에서는 원형 유니코드 리스트를 쓰면 충돌이 없다.

## AskUserQuestion 매핑

ⓐ 객관식 → `options` 배열, 라벨은 `"1"`, `"2"`, `"3"`, `"4"` (최대 4개까지가 도구 한도). "기타" 는 도구의 멀티-선택 문법으로 표현하거나, 자유 응답 후속 질의로 분리.
ⓑ 서브 옵션 있는 객관식 → 옵션 본문에 `1-a / 1-b` 라벨을 본문 텍스트로 표기.
ⓒ 자유 응답이 본질이면 도구 사용을 피하고 평문으로 질의.

## 예시 — 잘 된 형태

```
질의: 인증 모드를 정해야 합니다.

백엔드 게이트웨이가 두 흐름을 동시에 지원할 수는 있지만 초기 출시는 하나만 잡고 가는 편이 깔끔합니다. 운영 비용과 리프레시 토큰 처리 방식이 가장 큰 분기점입니다.

선택지:
1. 세션 쿠키 (서버 상태 보유)
2. JWT (무상태)
3. OAuth2 위임 (외부 IdP)
4. 다음 단계까지 보류
```

서브 옵션 예:

```
1. 세션 쿠키
   1-a. 메모리 저장소 (단일 인스턴스, 빠른 시작)
   1-b. Redis 저장소 (수평 확장 가능)
2. JWT
3. OAuth2 위임
```

설명 리스트 예 (질문 아님):

```
이 결정이 영향 주는 곳:
ⓐ 로그인 핸들러의 응답 헤더 형식
ⓑ 프론트엔드의 토큰 보관 위치
ⓒ E2E 테스트의 세션 갱신 시나리오
```

## 안티 패턴

ⓐ 한 메시지에 두 개 이상의 질문 — "X 와 Y 는 어떻게 할까요?"
ⓑ 6개 이상의 객관식 — 차원이 섞여 있다는 신호.
ⓒ 객관식 보기에 알파벳 — 사용자가 답할 때 형식이 흐려진다.
ⓓ 제안과 선택이 한 블록 — 사용자가 무엇에 답하는지 헷갈린다.
ⓔ 두괄식 누락 — 사용자가 끝까지 읽어야 핵심이 나오는 질문.
