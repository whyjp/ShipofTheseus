# 인터뷰 컨벤션

## 한 줄 요약
**모든 사용자 질의는 두괄식 + 한 번에 하나 + 객관식 *보기* 4개 이하 + 보기는 무조건 숫자 + 제안과 선택을 같은 블록에 섞지 않는다.** 이 규칙은 본 하네스가 사용자에게 던지는 모든 질문에 예외 없이 적용된다. **누적 질문 *수* 는 무한** — 4 는 *한 질문의 옵션 수* 한도 (`AskUserQuestion` 도구 한도) 이지 *질문 갯수* 한도가 아니다. 이 둘을 혼동하면 의도 추출이 망가진다 (사용자 v0.2.2 외부 테스트 회귀).

## 상세

### 1. 두괄식
질의의 첫 문단은 **요약 한 문장**. 무엇을 묻고, 왜 묻는지 한 줄로 결론. 배경·트레이드오프·고려사항은 그 다음 문단으로 분리한다. 사용자가 첫 줄만 읽고도 답할 수 있어야 한다.

### 2. 1회 1질의 (단, 누적 질의 수에 한도 없음)
한 호출에 한 질문만 — 답을 받은 뒤 다음. 단, **누적 질의 수에는 한도가 없다.** 사용자의 의도 누락, 진행 중 발생할 엣지 케이스, 사용자가 인지 못 한 예외를 사전에 잡아내기 위해서라면 10개·20개 질의도 정상이다. 묶음으로 한 번에 던지지 말고 차례로 풀어 묻는다 — 묶음 질문은 첫 답이 다음 답을 흐린다.

### 2-1. 엣지/예외 사전 확인 (페이즈 01)
의도 추출 단계의 마인드맵에서 의도적으로 부풀린 "예외/엣지" 가지를 사용자에게 보여주고, 항목별로 "다뤄야 함 / 무관함 / 추가 필요 / 모름" 4개 보기로 차례로 묻는다. ([`diagrams.md`](diagrams.md) 참조)

### 2-2. 확증 회귀 (Confirmation Recursion) — 인지 트릭
사용자가 자신의 의도를 *진짜로* 인지한 채 답하고 있는지 검증하기 위해, **같은 의미의 질문을 다른 표현·다른 관점으로 두 번** 묻는다. 인지·심리 검사의 reliability check 와 같은 원리.

ⓐ **숨김 회귀**: 페이즈 01 에서 질문 A 로, 페이즈 04 에서 같은 의미의 A' 로 묻되 어휘·문맥·보기 순서를 바꿔 같은 질문으로 보이지 않게 한다. 사용자가 자연스럽게 답할 수 있도록.
ⓑ **노출 회귀**: 두 답이 *다르면* 페이즈 05 에서 두 답을 나란히 보여주고 "어느 쪽이 진의입니까" 를 객관식으로 묻는다. 이때는 트릭을 숨기지 않고 정확히 드러낸다 — 사용자가 자기 답의 일관성을 자각하도록.
ⓒ **세분화**: 노출 회귀에서도 사용자가 모호한 답을 하면, 같은 의도를 더 작은 차원으로 쪼개 객관식 *보기* 4개 이하로 다시 묻는다 — 한 번에 한 차원씩. (분해된 질문 *갯수* 는 차원 수만큼 — 한도 없음.)

#### 회귀 짝의 예

| 페이즈 01 (자연 표현) | 페이즈 04 (전환 표현) |
| -------------------- | -------------------- |
| "사용자가 로그인 후 무엇을 가장 먼저 보길 원합니까?" | "로그인 직후 화면의 1순위 정보는?" |
| "이중 결제는 어느 쪽이 우선입니까, 거부 vs 보류?" | "동시에 같은 결제가 두 번 들어오면 시스템은 무엇을 합니까?" |

#### 일관성 마크

clarifier 가 `intent/04-answers.md` 의 각 답에 다음 메타를 부착:

```yaml
- question: "..."
  answer: "..."
  paired_with: "Q-01-3"            # 페이즈 01 의 어떤 질문과 짝인지
  consistency: "match" | "drift" | "contradict"
  drift_note: "사용자가 페이즈 01 에서 '거부' 답, 페이즈 04 에서 '보류' 답 — 노출 회귀 필요"
```

`drift` 또는 `contradict` 가 하나라도 있으면 페이즈 05 (비평) 에 *반드시* 노출 회귀 질문을 포함.

#### 트릭의 윤리적 가드

ⓐ 사용자에게 *불리한 정보를 빼는* 용도로 쓰지 않는다 — 의도 일관성 검증 한정.
ⓑ 사용자가 트릭을 알아채면 즉시 정직하게 "확증을 위해 같은 의미를 다른 표현으로 두 번 물었다" 고 인정한다.
ⓒ 회귀 짝은 페이즈 13 핸드오프 문서의 "결정 이력" 에 모두 기록 — 추적 가능성.
ⓓ 동의 없는 심리적 조작 금지 — 본 트릭의 목적은 *사용자의 자기 인지를 돕기* 위한 것.

#### 모호한 답의 세분화 절차

페이즈 04/05 에서 사용자가 회귀 짝에 일관되지 않거나 모호한 답을 하면:

① 같은 의도를 직교 차원 2~3 개로 분해.
② 차원마다 객관식 *보기* 4개 이하로 차례 질의 (차원 수만큼 질문 — 갯수 한도 없음).
③ 답을 합쳐 원 의도의 정밀한 좌표 도출.
④ 도출된 좌표를 다시 사용자에게 한 줄로 보여주고 "이게 맞으신가요" 4개 보기로 최종 확인.

예 (이중 결제 모호 답에 대한 분해):

```
질의 (차원 1/3): 동시 결제 두 건의 *시간 차* 가 1초 미만일 때, 두 번째는?

선택지:
1. 즉시 거부 (idempotency 우선)
2. 1초 대기 후 첫 거래 종료 확인 (보류)
3. 큐에 적재 후 운영자 알림
4. 모름 — 다음 차원 보고 결정

질의 (차원 2/3): 두 건의 *금액이 다를 때* 는?
...

질의 (차원 3/3): 두 건이 *다른 채널* (예: 웹 + 앱) 에서 들어오면?
...

질의 (종합): 위 답을 종합하면 — "1초 미만 시간차 + 동일 채널은 즉시 거부, 다른 채널은 1초 대기" — 이 정책이 맞으십니까?
1. 맞음 — 이대로 진행
2. 일부 수정 — 어느 차원
3. 다시 처음부터
4. 더 작은 차원으로 분해 필요
```

### 3. 객관식 *보기* 4개 이하 (질문 갯수와 무관)

**한 질문의 옵션 수 한도 = 4** (`AskUserQuestion` 도구 한도). 5번째 보기가 필요하면 잘못 모델링한 것 — 비슷한 옵션을 합치거나, 직교 차원으로 쪼개 두 단계로 나눠 묻는다. "기타" 는 4번째 옵션으로 두고 *자유 응답 후속 질의* 로 분리한다.

> ***4 는 옵션 수 한도지 질문 수 한도가 아님*** — 페이즈 04 인터뷰가 *총 4 질문* 으로 끝나는 게 정상 흐름이 아니다. NFR 항목별 1 질의 + Q-G1 그레이드 + Q-D1~D8 사전 위임 + 의도 모호 해소 + 회귀 짝 = *수십 개 질의* 가 정상. *옵션 한도 ≠ 질문 한도* (§2 의 *누적 질의 수 무한* 룰과 정합).

### 4. 보기는 무조건 숫자
객관식 선택지는 `1`, `2`, `3`, `4`. 알파벳·기호 사용 금지. 사용자가 "3" 이라고만 답해도 명확해야 한다. (Q-G1 그레이드 확정 같이 *5 등급* 인 예외 질의는 옵션 4 + 자유 응답 후속으로 분해 — `AskUserQuestion` 옵션 5번째 자리 없음.)

### 5. 서브 옵션은 `숫자-알파벳`
숫자 보기 안에 더 세분이 필요하면 `1-a`, `1-b`, `1-c`. 깊이는 한 단계까지만 — 더 깊어지면 별개 질의로 분리한다.

### 6. 순차 리스트(선택지 아님)는 원형 유니코드
질문이 아닌 **설명 안의 순차 항목** — 단계, 근거 나열, 예시 — 은 원형 알파벳 우선:

ⓐ 첫 항목
ⓑ 두 번째
ⓒ 세 번째

숫자가 꼭 필요하면 원형 숫자로:

① 첫 단계
② 두 번째 단계
③ 세 번째 단계

순수 텍스트 숫자(`1.`, `2.`)는 객관식 보기에만 쓴다 — 두 용도가 시각적으로 섞이면 안 된다.

### 7. 제안과 선택의 분리
"이런 안이 가능하다(제안)" 와 "둘 중 무엇으로 갈까요?(선택)" 를 같은 블록에 섞지 않는다. 문단·헤더로 분리하고, 선택 블록만 객관식 형식을 쓴다. 제안 단락에서는 원형 유니코드 리스트를 쓰면 충돌이 없다.

## AskUserQuestion 매핑

ⓐ 객관식 → `options` 배열, 라벨은 `"1"`, `"2"`, `"3"`, `"4"` (최대 4개까지가 도구 한도). "기타" 는 도구의 멀티-선택 문법으로 표현하거나, 자유 응답 후속 질의로 분리.
ⓑ 서브 옵션 있는 객관식 → 옵션 본문에 `1-a / 1-b` 라벨을 본문 텍스트로 표기.
ⓒ 자유 응답이 본질이면 도구 사용을 피하고 평문으로 질의.

## 예시 — 잘 된 형태

```
질의: 인증 모드를 정해야 합니다.

백엔드 게이트웨이가 두 흐름을 동시에 지원할 수는 있지만 초기 출시는 하나만 잡고 가는 편이 깔끔합니다. 운영 비용과 리프레시 토큰 처리 방식이 가장 큰 분기점입니다.

선택지:
1. 세션 쿠키 (서버 상태 보유)
2. JWT (무상태)
3. OAuth2 위임 (외부 IdP)
4. 다음 단계까지 보류
```

서브 옵션 예:

```
1. 세션 쿠키
   1-a. 메모리 저장소 (단일 인스턴스, 빠른 시작)
   1-b. Redis 저장소 (수평 확장 가능)
2. JWT
3. OAuth2 위임
```

설명 리스트 예 (질문 아님):

```
이 결정이 영향 주는 곳:
ⓐ 로그인 핸들러의 응답 헤더 형식
ⓑ 프론트엔드의 토큰 보관 위치
ⓒ E2E 테스트의 세션 갱신 시나리오
```

## 안티 패턴

ⓐ 한 메시지에 두 개 이상의 질문 — "X 와 Y 는 어떻게 할까요?"
ⓑ 6개 이상의 객관식 — 차원이 섞여 있다는 신호.
ⓒ 객관식 보기에 알파벳 — 사용자가 답할 때 형식이 흐려진다.
ⓓ 제안과 선택이 한 블록 — 사용자가 무엇에 답하는지 헷갈린다.
ⓔ 두괄식 누락 — 사용자가 끝까지 읽어야 핵심이 나오는 질문.

## PRD/스펙 입력 처리 (v0.4.0 — prd-handling.md 흡수)

> **중복 제거 (PR-13):** 기존 별도 컨벤션 `prd-handling.md` 가 본 절로 흡수. PRD 처리는 인터뷰의 *특수 모드* 라 별도 컨벤션 분리 비정합. 28 컨벤션 → 27.

사용자가 PRD/스펙 문서를 첨부했어도 본 페이즈 (Phase 04) 의 *모든 인터뷰 항목* 은 생략 안 됨. 이유:

ⓐ PRD 는 *작성된 시점의 의도* 일 뿐, 본 호출 시점에 *여전히 유효* 인지 미보장.
ⓑ PRD 추출값과 사용자 *현재* 답이 일치해도 *명시 확정* 이 의도 봉인의 시각 표시.
ⓒ self_lint C33 가 PRD 첨부 시에도 모든 답에 `user_explicit_confirmation: true` + `confirmed_at` timestamp 의무화.

PRD 추출값은 객관식의 *1번 보기* 로 제안 — 사용자가 빠른 1 클릭 확정. 4 보기 객관식 룰 위반 아님 — 1번이 *PRD 매핑*, 2~4번이 *PRD 와 다른 답* 후보, 자유 응답이 본질이면 도구 없이 평문으로 보강.

각 답에 `user_explicit_confirmation: true` + `prd_evidence_cited: true` + `confirmed_at` timestamp 가 frontmatter 에 박힘.

### PRD 인터뷰 안티 패턴

ⓐ PRD 가 충실하다는 이유로 인터뷰 스킵 — `intent/04-answers.md` 가 비면 페이즈 05 진입 거부.
ⓑ 1번 보기 (PRD 매핑) 만 자동 답 — 사용자 명시 클릭 없이 진행 금지.
ⓒ `confirmed_at` timestamp 누락 — self_lint C33 fail.

(참고: 본 절 흡수 전 별도 파일 `prd-handling.md` 가 있었으나 PR-13 에서 삭제. self_lint C2 인덱스 정합은 자동 갱신 — 컨벤션 디렉터리에 실제 존재하는 파일만 검증.)
