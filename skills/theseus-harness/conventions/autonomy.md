# 자율성 컨벤션 (Autonomy First)

## 한 줄 요약
**페이즈 04 (사용자 질의) 가 본 하네스의 *유일한* 사용자 인터럽트 지점이다.** 인터뷰 종료 후 핸드오프(페이즈 13) 까지 사용자 인터럽트는 절대 없다. 모든 후속 결정 — 회귀 권고, 경쟁 머지, 천정 도달 시 임계 조정, 정체 누적 시 행동, 자율 패키지 업데이트 — 은 페이즈 04 의 *사전 위임 카탈로그* 답에 따라 자율 적용된다.

## 핵심 원칙 — "인터뷰 후 인터럽트 0"

a- **의도는 스펙을 고정하지 않는다.** 페이즈 04 답조차 *해석된 의도* 의 부산물 — 진행 중 모든 수정은 자율.
b- **사용자 단 한 번 호출** — 페이즈 04 에서 의도 + 스택 + 리소스 프로파일 + NFR + 사전 위임 정책 모두 한 번에 결정.
c- **이후 모든 ack 는 사전 위임 카탈로그 매핑** — 페이즈 11 회귀 권고, 페이즈 10 천정 도달, lessons.md 의 3 회 누적 rewrite, competition.md 의 비즈니스 함의 경쟁 모두 *페이즈 04 답이 곧 답*.
d- **유일한 예외** — 페이즈 04 답으로 *해소 불가능한 새 정보* 가 발견된 경우 (예: 의도 자체가 모순). 이때만 정지하고 사람에게 보고. 그 외는 무한 자율.

## 사전 위임 카탈로그 (페이즈 04 의 마지막 6 질의)

본 하네스는 페이즈 04 끝에 다음 6 질의를 *반드시* 진행한다. 답이 없으면 페이즈 05 진행 불가.

### Q-D1. 회귀 (페이즈 11) 권고 자동 적용 정책

회귀 권고는 *항상 자율 적용* — 본 하네스의 핵심 원칙 (페이즈 04 외 인터럽트 0). Q-D1 답은 자율 처리의 *전략 선택* 이지 정지 옵션 아님.

```
질의: 회귀 권고 적용 전략?

선택지:
1. 모든 권고 자동 적용 (revert / re-architect / accept) — default
2. revert 우선 시도, 실패 시 re-architect 자동 fallback — 점진 전략
```

### Q-D2. 경쟁 resolve (competition.md) 자동 적용 정책

경쟁 resolve 는 *항상 자율* — 비즈니스 함의 경쟁이라도 점수 + 차원 분석으로 자동. Q-D2 답은 자율 *전략 선택*.

```
질의: 경쟁 resolve 적용 전략?

선택지:
1. 점수 차 + 차원 분석 자율 resolve (Δ ≥ 0.05 SELECT / Δ < 0.02 AUTO_MERGE / 중간 차원 분석) — default
2. 모든 경쟁 강제 머지 (점수 차 무관, 차원별 강점 합산) — 합쳐서 모두 살림 모드
```

### Q-D3. 천정 도달 (resources.md) 시 자동 임계 조정 정책

```
질의: NFR 차원이 리소스 천정에 도달 (avg ≥ 추정 천정의 90%) 시?

선택지 (모두 자율 적용, 정지 없음):
1. 권고 임계로 자동 조정 (측정 평균 × 1.05, 안전 여유 5%) — default
2. 도메인 단순화 (DB 캐시 / 인덱스 / 비동기화) 자동 시도 후 1 fallback
3. 정체 수용 — 게이트 영구 fail 로 표시하고 다음 페이즈 자율 진행
```

**v0.4.0 sub-option (보조 천정 흡수):**
- `1-aux` / `2-aux` — 위 1/2 답 + [`resources.md`](resources.md) "Opt-In 보조 천정" 활성. config.toml `[supplementary_ceiling]` 정합.
- 답이 sub-option 이면 본 하네스는 *임계 점수 + wall-clock + token + iter cap* 4 차원으로 정지 조건 검사 — 어느 하나라도 도달하면 `on_breach` 정책 발동.

### Q-D4. 정체 누적 (lessons.md) 시 정책

```
질의: 같은 차원 3 회 연속 rewrite 후에도 정체면?

선택지:
1. 페이즈 06 (계획) 부터 자동 재시작 — 다른 분할 시도, 무한 자율
2. 임계 자동 완화 (현재 측정값 × 1.05) 후 진행
3. 정체 수용 — 해당 차원 게이트 비활성, 다음 페이즈 진행
```

### Q-D5. 자율 패키지 업데이트 (stack.md) 정책

```
질의: 환경 점검에서 권고 미만 버전 발견 시?

선택지:
1. asdf/nvm/goenv 안에서 자동 업데이트 (시스템 영향 없음)
2. 시스템 패키지 매니저로 자동 업데이트 (sudo 필요, 영향 큼)
3. 업데이트 없이 현재 버전으로 진행 (호환성 위험 본인 부담)
```

### Q-D7. 체크포인트 회귀 + 멀티버스 정책 ([`checkpoints.md`](checkpoints.md))

```
질의: 실패 시 체크포인트 회귀 정책?

선택지:
1. 자동 회귀 + 멀티버스 자동 분기 (default — 닥터 스트레인지 모드, 가장 자율적)
2. 자동 회귀만 (멀티버스는 사용자 명시 시에만)
3. 회귀 없이 항상 fresh 시작 (단순, 학습 누적 없음)
```

### Q-D8. Verification Commands — 외부 완료 판정 (oh-my-ralph 차용, v0.3.0 신규)

본 하네스의 6 차원 rubric 가중평균 ≥ 0.999 는 *내부 정합* 측정이다 — 마크다운 산출물·테스트 통과율·SOLID 정합. 그러나 *외부에서 보아 완료* 인지는 별 문제 — 사용자 도메인의 acceptance test 가 통과해야 한다. [team-attention/oh-my-ralph](https://github.com/team-attention/oh-my-ralph) 의 *Verification Commands* 룰을 차용 — 사용자가 직접 실행 가능한 검증 스니펫을 제시해야 페이즈 05 (비평) 진입 허용.

```
질의: 본 작업의 *완료* 를 어떻게 검증하는가?

선택지:
1. CI 명령 그대로 사용 — 자유 응답으로 명령 한 줄 paste (예: `npm test && npm run typecheck && npm run lint`)
2. 새 검증 스니펫 작성 — 자유 응답으로 bash 블록 직접 입력 (acceptance criteria 별로 한 줄씩)
3. 자동 검증 불가 — 수동 검증만 (`manual_only: true`). 페이즈 09 게이트 의 `e2e_pass` 차원을 경고로 강하 (cap 0.95) + 핸드오프에서 외부 검증 책임 명시.
```

답이 1/2 면 사용자가 입력한 명령들을 `intent/04-verification.md` 의 `Verification Commands` 블록에 기록 + acceptance criteria `[SC-N]` 매핑 + frontmatter 에 `commands_count: <N>` + `manual_only: false` + `entry_blocked: false`. 답이 3 이면 같은 파일의 `Manual Verification` 섹션에 수기 절차 기록 + frontmatter `commands_count: 0` + `manual_only: true` + `entry_blocked: false`. **답 누락 또는 답 1/2 인데 Verification Commands 블록 비었으면 frontmatter `entry_blocked: true` 박힘** — [`agents/critic.md`](../agents/critic.md) 가 본 frontmatter 검사 후 페이즈 05 진입을 SkillEntryError 로 거부. self_lint C36 가 phases 04/05 + clarifier 의 Q-D8 wiring (entry_blocked / manual_only / oh-my-ralph 키워드) 일관성 검증.

답을 `intent/04-autonomy.md` 에 9 줄 표로 기록 (Q-D1 ~ Q-D9). 이후 모든 후속 페이즈는 본 답을 *유일한 ack source* 로 사용 — 추가 질의 금지.

### Q-D9. Runtime Prerequisite — 실행 가능 사전조건 (v0.7.0 신규, [`runtime-prereq.md`](runtime-prereq.md))

본 하네스의 게이트가 lint + 단위 테스트만 보장하면 *실 부팅* 에서 missing env / port 충돌 / DB connect 실패로 코드가 사용자 손에서 즉시 죽는다. Q-D8 이 *어떻게 검증하나* 를 정한다면 Q-D9 는 *어떻게 부팅하나* 를 정한다 — 두 답이 합쳐 코드-실행가능 까지를 한 약속에 묶는다.

```
질의: 본 작업의 *런타임 사전조건* 을 어떻게 처리하시겠습니까?

자동 추정: 외부 서비스 키워드 N개 감지.

선택지:
1. 실 env 값 paste (gitignored 별도 파일 즉시 봉인 + sha256 hash 만 audit)
2. `.env.template` 만 자동 생성 (사용자 후속 수기 채움)
3. 모든 외부 의존을 mock / disable 부팅 (dev profile)
4. 외부 의존 없음 (순수 로컬 / in-memory)
```

답에 따라 frontmatter 매핑:

| Q-D9 답 | env_satisfied | secrets_count | mode | entry_blocked |
| ------ | :-----------: | :-----------: | ---- | :-----------: |
| 1 | true | >0 | real | false |
| 2 | pending | >0 | template | false |
| 3 | true | 0 | mock | false |
| 4 | true | 0 | none | false |
| (미응답) | false | — | — | true |

답 1·2 동시 가능 (실 키 + template 둘 다). 답 3·4 단독.

본문은 `intent/04-runtime-prereq.md` 에 기록 (Q-D8 의 `intent/04-verification.md` 와 동일 패턴). 답이 1·2 인데 `.env.template` 비어있으면 frontmatter `entry_blocked: true` 박힘. 페이즈 09 게이트 7 (env-satisfied + 실 실행 1회) 가 본 답을 입력으로 부팅 검증.

**보안 — 자율 처리**: 실 secret 의 git 커밋이 감지되면 본 하네스가 자율로 *거부* + `.gitignore` 자동 추가 + handoff/13 사후 보고. 사용자 ack 호출 없음 (페이즈 04 외 인터럽트 0).

## 자율 결정의 사후 리뷰 가능성

모든 자율 결정은 다음 위치에 기록 — 사용자가 *핸드오프 후* 회수해 검토 가능 (인터뷰 중에는 회수 없음, 핸드오프 후만):

a- 결정 자체: 해당 페이즈 산출물의 본문.
b- 결정의 사유: 점수 + 차원별 분석 + 사전 위임 답 인용.
c- 결정의 대안: 경쟁이라면 패자 후보 디렉터리.
d- 결정 시각 + 책임 에이전트: frontmatter.

## 인터뷰 중 마지막 안전 장치 — "해석 불가" 신호

페이즈 05 ~ 13 중 *유일하게* 정지 + 사람에게 보고하는 케이스는 다음 둘:

a- **의도 자체 모순** — 페이즈 04 답들이 서로 모순 (예: "성능 지향" + "임계 무한 완화"). 본 하네스가 해석 불가능하면 정지.
b- **사전 위임 카탈로그 답 누락** — Q-D1 ~ Q-D6 중 하나라도 답이 없으면 페이즈 05 진입 불가, 페이즈 04 로 돌아가 답을 받음.

이 둘 외에는 인터럽트 0. 무한 자율.

## 자율 결정 신뢰도 가드

자율 결정의 *오결정* 위험을 줄이는 메커니즘 (이전 컨벤션 그대로 유지):

a- 점수 차 임계 — 경쟁 resolve.
b- DIP 단독 hard cap — SOLID 의 가장 중요한 원칙은 자율도 무시 불가.
c- 회귀 임계 — 자율 결정 후 점수 0.05 이상 하락 시 즉시 페이즈 11.
d- 확증 회귀 — interview.md 의 paired_with/consistency 추적.
e- 사후 리뷰 알림 — 핸드오프 (페이즈 13) 에 자율 결정 목록과 대안 비교 포함.

## 사용자에게 알리는 자율 결정 보고 형식

Q-D6 답에 따라 결정 보고 빈도가 다르지만, 본 하네스는 *침묵하지 않는다*. timing 헤더 라이브 보고:

```
스프린트 NN — 점수 0.997 (직전 0.991). 누적 41분. 자율 결정: 천정 도달 (p99 측정 142ms / 추정 천정 150ms 의 94.7%) → Q-D3.1 매핑 → 임계 142×1.05=149ms 자동 조정. 다음 스프린트 진행.
```

자율 결정의 사후 회수는 핸드오프 후. 인터뷰 종료 → 핸드오프 사이는 정지 명령 없음 (사용자가 외부에서 프로세스 자체를 kill 하는 방법 외).

## 안티 패턴

a- **인터뷰 종료 후 사용자 ack 호출** — 본 컨벤션의 핵심 위반. 모든 ack 는 페이즈 04 사전 위임 카탈로그에 매핑되어야 함.
b- 사전 위임 카탈로그 답 무시 — Q-D1~D6 답이 명확한데 추가 질의로 확인하면 사용자 신뢰 잃음.
c- 자율 결정을 *침묵 속에* 진행 — 라이브 보고 누락. 인터럽트 ≠ 침묵.
d- 점수 근거 없이 자율 결정 — 모든 자율 결정은 *왜* 가 산출물에 명시.
e- "해석 불가" 안전 장치 남발 — 진짜 모순일 때만 정지, 평범한 분기는 사전 위임 답 따라 자율.

| 회귀 권고 (revert/re-architect/accept) | **사전 위임 시 자율** | 위임 없으면 ack |
| 자율 패키지 업데이트 | 사용자 사전 동의 후 자율 | 환경 영향 |
| 페이즈 재실행 (verdict revise) | **자율** (3 회 캡) | 명세 흐름 |
| 산출물 frontmatter / 핑거프린트 | **자율** | 기계 작업 |
| 점수 0.999 미달 → 다음 스프린트 | **자율** | 명세 흐름 |
| 점수 0.05↓ 회귀 → 페이즈 11 발동 | **자율** | 명세 흐름 |
| 페이즈 11 권고 적용 (revert/re-architect) | 사전 위임 시 자율 | 코드 변경 위험 |
| 경쟁 머지/우승자 결정 | **자율** (resolve alg.) | competition.md §4 |
| 한 번도 본 적 없는 새 컨벤션 추가 | 사용자 ack | 명세 변경 |

## 자율 결정의 사후 리뷰 가능성

모든 자율 결정은 다음 위치에 기록된다 — 사용자가 어느 시점이든 회수해 검토 가능:

a- 결정 자체: 해당 페이즈 산출물의 본문 (예: `intent/05-decisions.md`, `sprints/NN/bisect.md`).
b- 결정의 *이유*: 같은 산출물의 "사유" 섹션 + 점수 근거.
c- 결정의 *대안*: 경쟁이라면 패자 후보 디렉터리(`competitions/.../losers/`) 에 보존.
d- 결정의 *시각*: frontmatter 의 `produced_at` + 시간 메타 헤더.
e- 결정의 *책임 에이전트*: frontmatter 의 `producer_agent`.

따라서 *어떤 자율 결정이라도 핑거프린트 + 시각 + 본문* 으로 추적 가능. 사용자가 사후에 "이건 다른 선택이었어야 했다" 고 지적하면 해당 페이즈만 재실행 (frontmatter 무결성 검증 후 [`contracts.md`](contracts.md) 의 재진입 룰).

## 자율 결정 신뢰도 가드

자율 결정의 *오결정* 위험을 줄이는 메커니즘:

a- **점수 차 임계** — 경쟁 resolve 는 점수 차가 0.02 미만이면 머지, 0.05 이상이면 우승자 채택, 그 사이는 차원별 분석. 임계 사이는 [`competition.md`](competition.md) §4 의 알고리즘으로 결정.
b- **DIP 단독 hard cap** — 후보가 DIP 위반이면 점수 무관하게 탈락. SOLID 의 가장 중요한 원칙은 자율 결정이라도 무시 불가.
c- **회귀 임계** — 자율 결정 이후 점수가 0.05 이상 떨어지면 즉시 페이즈 11 → 결정 자체를 재검토 (자율 위임이라도 회귀 후 재검토는 강제).
d- **확증 회귀** — [`interview.md`](interview.md) §2-2 의 paired_with/consistency 추적 — 사용자 의도와 자율 결정 사이의 정합 검증.
e- **사후 리뷰 알림** — 핸드오프 (페이즈 13) 에 자율 결정 목록과 대안 비교를 포함, 사용자가 "어떤 결정이 자율로 내려졌는지" 한눈에 봄.

## 자율 결정의 점수 디테일 확장

자율 판단의 검증 신뢰도를 높이려면 점수 차원의 디테일이 풍부해야 한다. 본 하네스의 점수는 다음 두 층:

a- **rubric 6 차원** ([`../scoring/rubric.md`](../scoring/rubric.md)) — correctness/scope_fit/solid/coverage/fe_be_parity/e2e_pass.
b- **sub_score 디테일** — 각 차원 내부의 세부 측정. 예:
  b-1 `correctness` = test_pass_rate × intent_fidelity (게이트 1 결과).
  b-2 `solid` = 모듈별 SOLID 통과 비율, DIP 위반은 cap 0.5.
  b-3 `coverage` = min(BE, FE) — avg 가 아닌 min 으로 패리티 강제.
  b-4 `fe_be_parity` = full/missing_one/missing_two/smoke_only/n/a 5 등급.
c- **메타 자기 평가** ([`../scoring/self_lint.py`](../scoring/self_lint.py) `--score`) — lint 35 + pytest 12 + sample 의 가중 평균. 임계 0.99999.

자율 결정이 신뢰받으려면 *결정 시점의 모든 점수* 가 산출물에 기록 + 핑거프린트로 봉인. 사후 회의에서 "그때 점수가 어땠지" 가 즉시 답 가능.

## 사용자에게 알리는 자율 결정 보고 형식

매 자율 결정 후 사용자에게는 *침묵하지 않는다* — [`timing.md`](timing.md) 의 라이브 보고에 한 줄 추가:

```
스프린트 NN 완료 — 점수 0.997 (직전 0.991). 누적 41분. 자율 결정: 경쟁 resolve (a 채택, Δ +0.06). 다음 스프린트 진행.
```

자율 결정의 사후 회수가 필요하면 사용자가 즉시 정지 명령 (예: "방금 자율 결정 다시 보자") — 정지는 항상 허용. 자율은 *사용자가 이의 없으면* 의 디폴트일 뿐, 항상 회수 가능.

## 사용자 사전 위임 메커니즘

페이즈 04 의 마지막 객관식 질의로 사용자가 자율 위임 수준을 결정:

```
질의: 본 하네스의 자율 판단 위임 수준을 정해주세요.

이번 작업 동안 페이즈 04 외에 사용자에게 추가 질의를 얼마나 자주 보낼지 결정합니다. 회귀 바이섹트와 자동 머지의 자율 처리 가능성에 영향.

선택지:
1. 최대 자율 (회귀·경쟁·머지 모두 자율, 핸드오프에서 일괄 사후 리뷰)
2. 회귀만 ack (회귀 권고는 사용자 ack, 그 외 자율)
3. 모든 분기 ack (보수적 — 회귀, 경쟁 resolve, 자율 업데이트 모두 ack)
4. 다음 스프린트마다 ack (스프린트 종료 시점마다 진행 여부 묻기)
```

답을 `intent/04-autonomy.md` 에 기록. 후속 페이즈는 이 위임 수준을 따른다.

## 안티 패턴

a- 자율 결정을 *침묵 속에* 진행 — 사용자가 무엇이 결정됐는지 모름. **모든 자율 결정은 라이브 보고 1 줄.**
b- 점수 근거 없이 자율 결정 — *왜* 그 후보가 우승했는지 산출물에 명시 안 됨. **점수 + 차원별 분석 필수.**
c- 위임 수준 무시하고 모든 분기를 자율 — 사용자가 "ack 만" 위임했는데 회귀를 자율 처리. **위임 수준은 페이즈 04 답이 source of truth.**
d- 자율 결정 후 산출물 누락 — 사후 리뷰 불가. **frontmatter + 핑거프린트 + 본문 누락 시 게이트 9 fail.**
